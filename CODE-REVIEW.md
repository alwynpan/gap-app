## CRITICAL

### 1. SQL Injection in Migration Script

**File:** `backend/src/db/migrate.js` (lines 72-79)

`generateMigrationSQL()` uses string interpolation to embed `adminUsername` and `passwordHash` directly into SQL. If `ADMIN_USERNAME` contains a single quote (e.g. `admin'; DROP TABLE users;--`), this is a textbook SQL injection. Use parameterized queries instead of template literals.

```js
// Current (vulnerable)
INSERT INTO users (...) VALUES ('${adminUsername}', 'admin@gap.local', '${passwordHash}', 1, true);

// Fix: use parameterized query via client.query(sql, params)
```

### 2. Duplicate bcrypt Libraries - Inconsistent Hashing

**File:** `backend/package.json` (lines 41-43, 45)

Three password hashing libraries ship simultaneously:

- `bcrypt` (native) — used by `migrate.js`
- `bcryptjs` (pure JS) — used by `User.js` model
- `fastify-bcrypt` (wrapper) — used by `rbac.js`

This creates inconsistency risks where hashes generated by one library might not be compatible with verification in another. Pick **one** and standardize.

### 3. RBAC `checkRole` Logic Is Inverted

**File:** `backend/src/middleware/rbac.js` (lines 13-27)

```js
const userRoleLevel = roleHierarchy[request.user.role] || 0;
const requiredLevel = Math.max(...requiredRoles.map((role) => roleHierarchy[role] || 0));

if (userRoleLevel < requiredLevel) {
  reply.code(403).send({ error: 'Forbidden: Insufficient permissions' });
  return false;
}
```

The function takes an array of `requiredRoles` but uses `Math.max` to find the **highest** required level. When called with `['admin', 'team_manager']` (e.g. in `users.js` line 13), it takes the max (admin=3), meaning a `team_manager` (level 2) is **denied** even though `team_manager` is explicitly in the allowed list. The intent is clearly "any of these roles should be allowed" but the implementation says "you need the highest-ranking role in the list." This should use `Math.min` or check if the user's role is **in** the allowed set.

### 4. Destructive Migration - Drops All Tables

**File:** `backend/src/db/migrate.js` (lines 25-28)

```sql
DROP TABLE IF EXISTS users CASCADE;
DROP TABLE IF EXISTS groups CASCADE;
DROP TABLE IF EXISTS roles CASCADE;
```

Running `npm run migrate` in production **destroys all data**. There's no migration versioning, no up/down support, no safety guard. A proper migration tool (e.g. `node-pg-migrate`, `knex`) should be used.

## HIGH

### 5. `pool.end()` Called After Migration Kills Shared Pool

**File:** `backend/src/db/migrate.js` (lines 110-111)

The same `pool` is **exported** and used by all models (`User.js`, `Group.js`, `Role.js`). If `migrate.js` is ever imported (not just run standalone) and `migrate()` is called, the pool shuts down and the entire app loses DB connectivity.

### 6. No Token Blacklisting / JWT Revocation

**File:** `backend/src/routes/auth.js` (lines 149-153)

Logout is a no-op server-side. A stolen/compromised JWT remains valid for the full 24h TTL. For a portal with RBAC, consider a token blacklist (Redis-backed) or shorter token expiry + refresh tokens.

### 7. Rate Limiting Configured But Plugin Not Registered

**File:** `backend/src/routes/auth.js` (lines 6-16) / `backend/src/server.js`

Routes define rate limit configs (`rateLimit: { max: 3, timeWindow: '1 minute' }`), but `@fastify/rate-limit` is **never registered** in `server.js`. The rate limit config objects are silently ignored. The package is in `package.json` but never `fastify.register()`'d.

### 8. `COALESCE` Cannot Set Fields to `null`

**File:** `backend/src/models/User.js` (lines 65-79), `backend/src/models/Group.js` (lines 24-35)

`COALESCE(null, group_id)` returns `group_id`, making it impossible to unassign a user from a group by passing `groupId: null`. The same applies to `Group.update`. A different update pattern is needed for nullable fields.

### 9. Frontend Token Stored in `localStorage` - XSS Risk

**File:** `frontend/src/context/AuthContext.jsx` (lines 54-55)

`localStorage` is accessible to any JavaScript running on the page. An XSS vulnerability would expose the JWT. `httpOnly` cookies are the safer alternative for token storage.

### 10. Nginx Missing Security Headers

**File:** `frontend/nginx.conf`

No security headers configured:

- No `X-Content-Type-Options: nosniff`
- No `X-Frame-Options: DENY`
- No `Strict-Transport-Security`
- No `X-XSS-Protection`
- No `Content-Security-Policy`
- No `Referrer-Policy`

The proxy blocks also don't forward `X-Real-IP` or `X-Forwarded-For`, breaking rate limiting and audit logging.

## MEDIUM

### 11. Duplicate `lint-staged` Keys in Both `package.json` Files

**Files:** `backend/package.json` (lines 17-34), `frontend/package.json` (lines 17-34)

Both files have the `lint-staged` key defined **twice**. JSON spec says last-key-wins, but this is clearly a copy-paste error and linters won't flag it.

### 12. `@fastify/postgres` Is a Dependency But Never Used

**File:** `backend/package.json` (line 39)

The app uses raw `pg.Pool` directly. This dependency is dead weight.

### 13. Inefficient Duplicate Name Check

**File:** `backend/src/routes/groups.js` (lines 108-109)

```js
const existingGroups = await Group.findAll();
const existingGroup = existingGroups.find((g) => g.name.toLowerCase() === name.toLowerCase());
```

This fetches **all** groups from the database to check for name uniqueness in memory. Should be a targeted query like `SELECT 1 FROM groups WHERE LOWER(name) = LOWER($1)`.

### 14. Graceful Shutdown Doesn't Close Connections

**File:** `backend/src/server.js` (lines 123-131)

Neither `SIGTERM` nor `SIGINT` handler calls `fastify.close()` or `pool.end()`. Active requests and DB connections are killed abruptly. The `fastify` instance isn't accessible in scope either.

### 15. `authenticate` Decorator Defined But Never Used

**File:** `backend/src/server.js` (lines 35-42)

This decorator is registered but no route uses it. Routes implement their own ad-hoc auth checks via `preHandler` instead.

### 16. Global Axios Headers Are a Shared Mutable State Hazard

**File:** `frontend/src/context/AuthContext.jsx` (lines 14-19)

Mutating `axios.defaults` is global. If another part of the app creates an axios instance or multiple AuthProviders exist, tokens leak across instances. Use an axios interceptor or create a dedicated axios instance.

### 17. `Users.jsx` and `Groups.jsx` Use `<a>` Instead of React Router `<Link>`

**Files:** `frontend/src/pages/Users.jsx` (line 77), `frontend/src/pages/Groups.jsx` (line 94)

Using `<a href="/dashboard">` causes a full page reload, losing React state. Should be `<Link to="/dashboard">`.

### 18. K8s `gap-service` Has Mismatched Backend Port

**File:** `k8s/gap-service.yaml`

The service has a `backend` port (3001) but its selector targets `gap-frontend` pods, which don't run on port 3001. Backend traffic via this service would fail.

### 19. K8s Ingress `tls: []` - TLS Declared But Empty

**File:** `k8s/gap-ingress.yaml` (line 60)

This declares TLS but configures nothing. Either remove it or configure a proper TLS secret. The `websecure` Traefik entrypoint annotation implies HTTPS is intended.

### 20. No `ADMIN_PASSWORD` Env Var in K8s Backend Deployment

**File:** `k8s/gap-backend-deployment.yaml`

The backend deployment has `JWT_SECRET` and `DB_PASSWORD` from secrets, but `ADMIN_PASSWORD` and `ADMIN_USERNAME` (needed by `migrate.js`) are not provided. Migration will fail with `process.exit(1)`.

## LOW

### 21. `/auth/me` Returns Stale JWT Claims, Not Fresh DB Data

**File:** `backend/src/routes/auth.js` (lines 155-177)

This returns whatever is baked into the JWT. If an admin changes a user's role or group, the user won't see the change until their token expires (24h). Should query the DB for fresh data.

### 22. Password Minimum Length Is Only 6 Characters

**File:** `backend/src/routes/auth.js` (lines 30-33)

NIST recommends at least 8 characters. No complexity requirements (uppercase, number, special char) either.

### 23. No Password Strength Validation on Admin-Created Users

**File:** `backend/src/routes/users.js` (lines 68-133)

`POST /users` (admin create) has no password length or format validation at all, unlike `/auth/register`.

### 24. Frontend Missing Error Boundary

No React error boundary component exists. An unhandled JS error in any component crashes the entire app with a white screen.

### 25. Dashboard "Quick Actions" Buttons Are Non-Functional

**File:** `frontend/src/pages/Dashboard.jsx` (lines 103-109)

The "View My Progress" and "View Assignments" buttons have no `onClick` handlers. They render but do nothing.

### 26. `handleAssignGroup` Has Dead Logic

**File:** `frontend/src/pages/Users.jsx` (lines 48-55)

```js
const handleAssignGroup = async () => {
  if (!selectedUser || selectedGroup === '') return;
  const groupId = selectedGroup === '' ? null : parseInt(selectedGroup);
```

The ternary on line 51 will never produce `null` because line 49 already returns early if `selectedGroup === ''`.

### 27. Backend Dockerfile Copies Source Before .dockerignore Check

**File:** `backend/Dockerfile` (line 11)

If `.dockerignore` is missing, `node_modules`, `.env`, test files, and other dev artifacts get copied into the production image. No `.dockerignore` was found in the repo.

### 28. No K8s `NetworkPolicy` - All Pods Can Talk To All Pods

No network policies exist. Any compromised pod can reach the database directly. Postgres should only be reachable from the backend pods.
